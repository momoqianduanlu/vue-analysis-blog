# patch

通过上一小节的分析我们知道，当我们通过 createComponent 创建了组件 VNode，接下来会走到 vm._update，执行 vm.__patch__ 去把 VNode 转换成真正的 DOM 节点。这个过程我们在前一章已经分析过了，但是是针对一个普通的 VNode 节点，接下来我们来看看组件的 VNode 进行patch 会有哪些不一样的地方。

首先来构建自己的demo工程目录，入口文件是App.vue，使用render函数进行编译，那么App.vue文件里面又有一些dom节点和子组件，下面我们就来分析父子组件是如何进行渲染的。

    export function createComponentInstanceForVnode (
  vnode: any, // 组件vnode
  parent: any, // activeInstance in lifecycle state // vm实例(会作为子组件的父vm实例)

  const options: InternalComponentOptions = {
    _isComponent: true,
    _parentVnode: vnode,
    parent

  const inlineTemplate = vnode.data.inlineTemplate

    createComponentInstanceForVnode 函数构造的一个内部组件的参数，然后执行 new vnode.componentOptions.Ctor(options)。

    这里的 vnode.componentOptions.Ctor 对应的就是子组件的构造函数，我们上一节分析了它实际上是继承于 Vue 的一个构造器 Sub，相当于 new Sub(options) 这里有几个关键参数要注意几个点，_isComponent 为 true 表示它是一个组件，parent 表示当前激活的组件实例(在当前这个case下，这个实例目前是 new Vue 返回的 vm 实例)，_parentVnode则表示是一个占位符vnode，也就是我们当前app.vue的parentVnode。

    最后走到了子组件的构造器new vnode.componentOptions.Ctor(options)，

  vm,
  el,
  hydrating
) {
  vm.$el = el;

  /* istanbul ignore if */
  if (process.env.NODE_ENV !== 'production' && config.performance && mark) {

      var name = vm._name;
      var id = vm._uid;
      var startTag = "vue-perf-start:" + id;
      var endTag = "vue-perf-end:" + id;

      mark(startTag);
      var vnode = vm._render();
      mark(endTag);

      mark(startTag);

      mark(endTag);

  } else {

        callHook(vm, 'beforeUpdate');
      }
    }

    这里的 _parentVnode 就是当前组件的父 VNode，而 render 函数生成的 vnode 是当前组件的渲染 vnode，vnode 的 parent 指向了 _parentVnode(占位符vnode)，也就是 vm.$vnode，它们是一种父子的关系。

    如何理解占位符vnode和渲染vnode？

    拿我们当前的demo来举列子，App.vue中有一个子组件Hello-world.vue，那么在App.vue中Hello-world这个标签只是用来占位，也就是说他此时就是一个占位符vnode，而真正的渲染vnode是要通过我们的render函数来将Hello-world.vue渲染成一个vnode。

    // vm此时是App.vue的实例
Vue.prototype._render = function (): VNode {
  const vm: Component = this
  const { render, _parentVnode } = vm.$options

  
  // set parent vnode. this allows render functions to have access
  // to the data on the placeholder node.
  vm.$vnode = _parentVnode
  // render self
  let vnode
  try {
    // 通过调用render方法生成App.vue的渲染vnode
    vnode = render.call(vm._renderProxy, vm.$createElement)
  } catch (e) {
    // ...
  }
  // set parent
  vnode.parent = _parentVnode
  return vnode
}

    我们知道在执行完 vm._render 生成 VNode 后，接下来就要执行 vm._update 去渲染 VNode 了。

    来看一下组件渲染的过程中有哪些需要注意的，vm._update 的定义在 src/core/instance/lifecycle.js 中：

    export let activeInstance: any = null
Vue.prototype._update = function (vnode: VNode, hydrating?: boolean) {
  const vm: Component = this
  const prevEl = vm.$el
  const prevVnode = vm._vnode
  const prevActiveInstance = activeInstance // 这里activeInstance是vm实例
  activeInstance = vm // 这里vm是App.vue组件的实例
  vm._vnode = vnode
  // Vue.prototype.__patch__ is injected in entry points
  // based on the rendering backend used.
  if (!prevVnode) {
    // initial render
    vm.$el = vm.__patch__(vm.$el, vnode, hydrating, false /* removeOnly */)
  } else {
    // updates
    vm.$el = vm.__patch__(prevVnode, vnode)
  }
  activeInstance = prevActiveInstance
  // update __vue__ reference
  if (prevEl) {
    prevEl.__vue__ = null
  }
  if (vm.$el) {
    vm.$el.__vue__ = vm
  }
  // if parent is an HOC, update its $el as well
  if (vm.$vnode && vm.$parent && vm.$vnode === vm.$parent._vnode) {
    vm.$parent.$el = vm.$el
  }
  // updated hook is called by the scheduler to ensure that children are
  // updated in a parent's updated hook.
}

    _update 过程中有几个关键的代码，首先 vm._vnode = vnode 的逻辑，这个 vnode 是通过 vm._render() 返回的组件渲染 VNode，vm._vnode 和 vm.$vnode 的关系就是一种父子关系，用代码表达就是 vm._vnode.parent === vm.$vnode。

    这个 activeInstance 作用就是保持当前上下文的 Vue 实例，它是在 lifecycle 模块的全局变量，定义是 export let activeInstance: any = null，并且在之前我们调用 createComponentInstanceForVnode 方法的时候从 lifecycle 模块获取，并且作为参数传入的。因为实际上 JavaScript 是一个单线程，Vue 整个初始化是一个深度遍历的过程，在实例化子组件的过程中，它需要知道当前上下文的 Vue 实例是什么，并把它作为子组件的父 Vue 实例。之前我们提到过对子组件的实例化过程先会调用 initInternalComponent(vm, options) 合并 options，把 parent存储在 vm.$options 中，

    接下来会执行到子组件的_patch，也就是说当我们执行完最外层App.vue的path以后会返回该组件的占位符vnode，占位符vnode执行完整个App.vue的初始化后才开始执行app.vue内部的子组件的patch。

    vm.$el = vm.__patch__(vm.$el, vnode, hydrating, false /* removeOnly */)

    if (isDef(vnode.componentInstance)) {
      initComponent(vnode, insertedVnodeQueue)
      insert(parentElm, vnode.elm, refElm)
      if (isTrue(isReactivated)) {
        reactivateComponent(vnode, insertedVnodeQueue, parentElm, refElm)
      }
      return true
    }
  }
}

      此时，我们可以认为当前场景在子组件Hello-world patch 之后，vnode是Hello-world的占位符vnode，vnode.componentInstance.$el是div.hello，也就是我们执行子组件patch之后的结果。

      然后他会将patch之后的结果赋值给我们的占位符vnode.elm。

      function initComponent (vnode, insertedVnodeQueue) {
    if (isDef(vnode.data.pendingInsert)) {
      insertedVnodeQueue.push.apply(insertedVnodeQueue, vnode.data.pendingInsert);
      vnode.data.pendingInsert = null;
    }
    vnode.elm = vnode.componentInstance.$el;
    if (isPatchable(vnode)) {
      invokeCreateHooks(vnode, insertedVnodeQueue);
      setScope(vnode);
    } else {
      // empty component root.
      // skip all element-related modules except for ref (#3455)
      registerRef(vnode);
      // make sure to invoke the insert hook
      insertedVnodeQueue.push(vnode);
    }
  }

      执行完initComponent以后到inset，此时，parentElm是div id=app的dom对象，vnode.elm是div.hello，也就是Hello-world往div id = app上面插入，完成了这次插入以后，那么App.vue也会往body上面插入，再执行一遍代码就可以了，

      在完成组件的整个 patch 过程后，最后执行 insert(parentElm, vnode.elm, refElm) 完成组件的 DOM 插入，如果组件 patch 过程中又创建了子组件，那么DOM 的插入顺序是先子后父。
